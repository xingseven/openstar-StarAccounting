## 3. 深度功能实现细节

### 3.1 APP 连接认证流程 (Secure Connection Flow)
**核心需求**: 
APP 端需输入部署服务器的公网 IP 和一次性验证码 (OTP) 才能建立连接。验证码需在服务端生成并校验，确保安全性。

**流程设计**:

1.  **生成验证码 (Web 端)**:
    *   用户进入 "连接管理" 页面。
    *   点击 "生成连接码" 按钮。
    *   **后端逻辑**: 
        *   生成 6 位随机数字 `otpCode`。
        *   创建 `AppConnection` 记录，设置 `expiresAt = now() + 5min`。
        *   返回 `otpCode` 和当前服务器 `publicIP` 给前端展示。
    *   **UI 展示**: 显示巨大的数字验证码和二维码 (包含 IP+Code 信息的 JSON)。

2.  **APP 端发起连接**:
    *   用户输入 (或扫码获取) 服务器 IP 和 验证码。
    *   APP 发送 POST 请求: `http://<ServerIP>/api/connection/verify`
    *   Body: `{ otpCode: "123456", deviceId: "uuid", deviceName: "Pixel 7" }`

3.  **服务端校验**:
    *   查找数据库中 `otpCode` 匹配且 `isVerified=false` 且 `expiresAt > now()` 的记录。
    *   **验证失败**: 返回 400 Invalid Code or Expired。
    *   **验证成功**: 
        *   更新记录: `isVerified = true`, `deviceId = ...`, `verifiedAt = now()`。
        *   生成并返回长期有效的 `AccessToken` (JWT) 给 APP。
        *   (可选) Web 端通过 WebSocket/轮询 收到 "新设备已连接" 通知。

### 3.2 智能还款联动系统 (Smart Repayment Linkage)
**核心需求**: 从账单流水中自动识别还款记录，并更新对应贷款的余额和期数。

**工作流设计**:

1.  **关键词配置 (Configuration)**:
    *   在创建/编辑 `Loan` 时，用户需填写 `matchKeywords`（如 `["微粒贷", "房贷", "车贷"]`）。
    *   系统也可提供预设关键词库（如 "借呗" -> `["蚂蚁借呗", "网商银行"]`）。

2.  **自动关联 (Auto-Link Logic)**:
    *   **触发时机**: 用户导入账单 (ETL) 或手动添加流水时。
    *   **匹配规则**:
        ```typescript
        // 伪代码逻辑
        function autoLinkTransaction(transaction: Transaction, userLoans: Loan[]) {
          // 1. 遍历用户所有 ACTIVE 状态的贷款
          for (const loan of userLoans) {
            // 2. 检查交易备注/商家名称是否包含贷款关键词
            const isMatch = loan.matchKeywords.some(keyword => 
              transaction.description.includes(keyword) || 
              transaction.merchant.includes(keyword)
            );
            
            // 3. 如果匹配且交易类型为支出
            if (isMatch && transaction.type === 'EXPENSE') {
              return {
                ...transaction,
                type: 'REPAYMENT', // 自动修正类型为还款
                loanId: loan.id
              };
            }
          }
          return transaction;
        }
        ```

3.  **数据联动 (Data Sync)**:
    *   **原子操作**: 使用 Prisma Interactive Transactions (`$transaction`) 确保数据一致性。
    *   **更新逻辑**:
        *   当插入一条 `REPAYMENT` 流水时：
            1.  `Loan.remainingAmount` -= `transaction.amount`
            2.  `Loan.paidPeriods` += 1 (可选：根据金额判断是否足额还款)
            3.  若 `Loan.remainingAmount <= 0`，自动更新 `Loan.status = PAID_OFF`。

### 3.3 数据清洗 ETL (Extract-Transform-Load)
**核心痛点**: 
*   **格式不统一**: 微信/支付宝的 CSV 文件通常包含大量非数据行（如个人信息、统计汇总）。
*   **列名识别**: 需要准确识别关键列，并跳过前面的 "垃圾行"。

**1. 智能表头识别算法 (Header Detection)**:
*   **策略**: 不要硬编码 "第 25 行开始"，而是逐行扫描，直到找到包含特定**特征列名**的行。
*   **特征列名**:
    *   **微信**: 包含 `["交易时间", "交易类型", "金额(元)"]`。
    *   **支付宝**: 包含 `["交易时间", "交易分类", "金额"]` 或 `["交易时间", "商家订单号"]`。
*   **实现步骤**:
    1.  使用 `PapaParse` 读取文件前 50 行 (Preview)。
    2.  对每一行进行关键词匹配。
    3.  锁定表头所在行号 (Header Index)。
    4.  从 `Header Index + 1` 开始解析正式数据。

**2. 字段映射表 (Field Mapping)**:

| 目标字段 (DB) | 微信 CSV 列名 | 支付宝 CSV 列名 | 处理逻辑 |
| :--- | :--- | :--- | :--- |
| **date** | `交易时间` | `交易时间` | 解析 `yyyy-MM-dd HH:mm:ss` |
| **category** | `交易类型` | `交易分类` | 优先使用导入值，后续支持清洗映射 |
| **merchant** | `交易对方` | `交易对方` | 去除 "扫二维码付款给" 等前缀 |
| **description** | `商品` + `备注` | `商品说明` + `备注` | 拼接存储，用于搜索 |
| **type** | `收/支` | `收/支` / `资金状态` | "支出"->EXPENSE, "收入"->INCOME, 其他->不导入 |
| **amount** | `金额(元)` | `收/支金额` | 去除 `¥` 符号，转 Decimal |
| **paymentMethod**| `支付方式` | `收/付款方式` | 记录资金来源 (如 "招商银行", "零钱") |
| **status** | `当前状态` | `交易状态` | **过滤**: 仅导入 `["支付成功", "交易成功", "已存入", "已取出"]` |
| **orderId** | `交易单号` | `交易订单号` | **去重关键**: 数据库设为 Unique，存在则跳过 |

**3. 导入流程**:
1.  **Upload**: 用户拖拽文件。
2.  **Analyze**: 前端 Web Worker 预读取，识别表头，展示 "预计导入 N 条数据"。
3.  **Map & Clean**: 应用上述映射规则，过滤无效/失败交易。
4.  **Review**: (可选) 展示前 5 条解析结果供用户确认。
5.  **Commit**: 批量提交到后端 API。

### 3.4 集中式主题管理策略 (Centralized Theme Management)
**核心决策**: 采用 **集中式主题包** 结构，而非分散式。

**理由**:
1.  **高内聚**: 一个主题的所有视觉资产（颜色、圆角、字体、布局配置）都在一个文件夹内，方便迁移和整体替换。
2.  **配置驱动**: 业务组件（如 `Sidebar`）应保持逻辑纯粹，通过读取“当前主题配置”来决定渲染样式，而不是在组件内写死 `if (theme === 'A')`。

**目录结构详解 (`src/themes/`)**:
*   `index.ts`: 导出所有可用主题列表。
*   `[theme-name]/`:
    *   `tokens.css`: 定义 CSS 变量 (Colors, Radius, Spacing)。
        ```css
        .theme-cyberpunk {
          --background: 224 71% 4%;
          --foreground: 213 31% 91%;
          --primary: 262 83% 58%;
        }
        ```
    *   `config.ts`: JS 配置对象，定义非 CSS 可控的逻辑。
        ```typescript
        export const config = {
          layoutType: 'sidebar-left', // 或 'navbar-top'
          chartPalette: ['#ff00ff', '#00ffff'], // 图表默认色盘
          sidebarStyle: 'glassmorphism' // 侧边栏风格变体
        }
        ```

**侧边栏实现策略**:
业务组件 `Sidebar.tsx` 不直接包含特定主题代码，而是作为**容器**：
```tsx
// components/shared/Sidebar/index.tsx
import { useTheme } from '@/hooks/useTheme';
import { SidebarClassic } from './variants/Classic';
import { SidebarModern } from './variants/Modern';

export function Sidebar() {
  const { config } = useTheme();
  
  // 根据主题配置动态选择布局组件
  switch (config.layoutType) {
    case 'modern': return <SidebarModern />;
    case 'classic': 
    default: return <SidebarClassic />;
  }
}
```
这样既保证了主题文件的集中管理，又实现了不同主题下布局结构的灵活性。

### 3.5 动态主题引擎 (Theme Engine)
**需求**: 用户选择一个主色，系统自动生成完整的色阶 (50-950)。

**算法逻辑 (`lib/theme-engine.ts`)**:
1.  **输入**: 用户选择 HEX 颜色 (e.g., `#3b82f6`).
2.  **转换**: HEX -> HSL (Hue, Saturation, Lightness)。
3.  **生成**: 保持 Hue (色相) 不变。
    *   **Lightness (亮度)**: 从 95% (50) 到 5% (950) 等差分布。
    *   **Saturation (饱和度)**: 在极亮/极暗处适当降低饱和度以提升可读性。
4.  **输出**: 生成 CSS 变量对象。
    ```css
    :root {
      --primary: 221.2 83.2% 53.3%;
      --primary-foreground: 210 40% 98%;
      --chart-1: ...; /* 自动适配图表颜色 */
    }
    ```
5.  **应用**: 通过 `style` 标签注入到 `<body>` 或使用 Tailwind 的 `Config` 动态更新。

### 3.6 复杂图表交互
**组件**: `Recharts` 封装
1.  **下钻 (Drill-down)**: 点击 "餐饮" 饼图扇区 -> 触发 API 查询该分类下的 "商家" 排名 -> 更新右侧柱状图。
2.  **缩放 (Zoom)**: 消费趋势折线图支持 `Brush` 组件，允许用户拖拽选择时间范围查看细节。
3.  **多维度切换**: 
    *   提供 `ChartSwitcher` 组件，允许用户在 "Bar", "Line", "Area" 之间切换。
    *   状态存储在 `UserSettings` 中，记住用户的偏好。

### 3.7 贷款计算器 (Loan Calculator)
虽然主要是记录，但需提供**试算功能**：
*   **等额本息 (Equal Principal & Interest)**: 
    $$ Monthly = [P \times R \times (1+R)^N] / [(1+R)^N - 1] $$
*   **等额本金 (Equal Principal)**: 
    $$ Monthly(n) = (P / N) + (P - (n-1) \times (P/N)) \times R $$
*   **功能**: 在前端实时计算，生成还款计划表预览，用户确认后保存为 `Loan` 记录。

### 3.8 图表与数据映射分析 (Chart Data Mapping)
为了确保图表能正确渲染，数据库必须提供相应的聚合查询能力。

| 模块 | 图表名称 | 图表类型 | 数据源需求 (Group By / Aggregation) | DB 字段依赖 |
| :--- | :--- | :--- | :--- | :--- |
| **消费** | **消费平台分布** | Pie Chart | `SUM(amount)` GROUP BY `platform` (微信/支付宝) | `platform`, `amount`, `type='EXPENSE'` |
| | **收支占比** | Donut | `SUM(amount)` GROUP BY `type` (INCOME/EXPENSE) | `type`, `amount` |
| | **消费商家排行** | Bar Chart (Top 10) | `SUM(amount)` GROUP BY `merchant` ORDER BY sum DESC | `merchant`, `amount` |
| | **消费趋势** | Line Chart | `SUM(amount)` GROUP BY `DATE_TRUNC('day', date)` | `date`, `amount` |
| | **消费类别堆积** | Stacked Bar | `SUM(amount)` GROUP BY `date`, `category` | `category`, `date` |
| | **热力图** | Heatmap | `SUM(amount)` GROUP BY `platform`, `category` | `platform`, `category` |
| **储蓄** | **存取记录** | Table/Line | `SELECT *` WHERE `category` IN ('理财', '存款') | `category`, `amount`, `paymentMethod` (资金流向) |
| **贷款** | **平台金额占比** | Pie Chart | `SUM(totalAmount)` GROUP BY `platform` FROM `Loan` | `Loan.platform`, `Loan.totalAmount` |
| | **已还/未还分析** | Grouped Bar | `SUM(paid)`, `SUM(remaining)` GROUP BY `platform` | `Loan.remainingAmount`, `Loan.totalAmount` |

## 4. 前端组件详细拆解

### 4.2 连接管理页 (`/connections`)
**功能**: 管理已授权的 APP 设备，生成新的连接验证码。

## 7. 业务域与边界上下文 (Domain Boundaries)

### 7.1 领域划分
*   **账户域 (Account Domain)**: 管理资金容器（银行卡、零钱、信用卡、投资账户）。
*   **交易域 (Transaction Domain)**: 管理所有流水与分类标签，是分析系统核心输入。
*   **贷款域 (Loan Domain)**: 管理借贷合同、还款计划、剩余应还、逾期状态。
*   **储蓄域 (Savings Domain)**: 管理目标、进度、里程碑、目标状态。
*   **连接域 (Connection Domain)**: 管理 Web 与 APP 的安全绑定。
*   **主题域 (Theme Domain)**: 管理主题配置、动态色阶与布局变体。

### 7.2 聚合根定义
*   **User**: 系统根聚合，拥有所有子域资源。
*   **Transaction**: 独立聚合根，支持与 Loan 的弱关联。
*   **Loan**: 独立聚合根，内部维护 `remainingAmount` 与 `paidPeriods` 一致性。
*   **AppConnection**: 一次性验证码与设备绑定聚合，具备强时效与不可复用约束。

### 7.3 一致性规则
*   同一用户下 `orderId` 必须唯一，防止重复导入。
*   还款联动必须在同一数据库事务中更新 Transaction 与 Loan。
*   OTP 验证成功后立即失效，禁止二次使用。

## 12. 贷款联动算法细化

### 12.1 匹配优先级
1.  备注关键词精确匹配（最高优先级）
2.  交易对方模糊匹配
3.  平台 + 固定还款日时间窗口匹配

### 12.2 多贷款冲突处理
当一条还款可能匹配多个 Loan：
*   先按关键词权重排序（完全匹配 > 前缀匹配 > 包含匹配）
*   再按 `remainingAmount` 接近度排序
*   最后按最近还款时间排序

### 12.3 联动事务伪代码
```typescript
await prisma.$transaction(async (tx) => {
  const transaction = await tx.transaction.create({ data: payload });
  const loan = await findBestMatchedLoan(tx, transaction);
  if (!loan) return;

  await tx.transaction.update({
    where: { id: transaction.id },
    data: { type: 'REPAYMENT', loanId: loan.id }
  });

  const remaining = max(loan.remainingAmount - transaction.amount, 0);
  await tx.loan.update({
    where: { id: loan.id },
    data: {
      remainingAmount: remaining,
      paidPeriods: loan.paidPeriods + 1,
      status: remaining === 0 ? 'PAID_OFF' : 'ACTIVE'
    }
  });
});
```

## 25. APP 同步协议与离线策略

### 25.1 Token 与设备绑定
*   APP Token 与 `deviceId` 强绑定。
*   同一 token 在不同设备使用时立即失效。
*   支持“踢下线”策略（撤销授权后立即失效）。

### 25.2 离线写入与冲突解决
*   APP 离线时写入本地队列（按时间排序）。
*   恢复网络后按队列顺序提交。
*   冲突策略:
    *   创建冲突: 以 `clientGeneratedId` 去重。
    *   更新冲突: 默认“服务器版本优先”，保留冲突副本供用户确认。

### 25.3 同步模式
*   全量同步: 首次登录。
*   增量同步: 基于 `updatedAt > cursor`。
*   拉取间隔: 前台 30s，后台 5min（可配置）。

## 29. 数据迁移与历史兼容策略

### 29.1 迁移原则
*   所有数据库结构变更必须通过 Prisma Migration。
*   禁止线上手工直接改表。
*   灰度发布时必须支持前后版本兼容至少一个发布周期。

### 29.2 历史数据兼容
*   对历史无 `orderId` 记录执行补偿标记。
*   对旧字段 `paymentMethod` 空值执行兜底分类。
*   对历史 `loan` 未联动流水运行离线重算任务。

### 29.3 回滚策略
*   应用回滚与数据库回滚分离设计。
*   高风险变更采用“双写 + 校验 + 切换”模式。

### 30.1 基础资源建议
