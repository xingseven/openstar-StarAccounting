# OpenStar 后端稳定性与总览性能优化开发文档

## 0. 文档版本

- 大版本：`v1.0`
- 小版本：`v1.0.0`
- 本次更新：收口总览页的数据请求链路，修复后端统计接口的账户范围与筛选遗漏，降低“后端反复重启后仍感觉很慢”的排查成本。

---

## 1. 背景

本轮问题表现为两类：

1. 登录或刷新后，前端偶发提示 `3006` 连接失败，用户感知为“后端又挂了”。
2. 即使后端在线，页面数据加载仍明显偏慢，尤其是总览页。

排查后确认，真正的问题不只在“服务是否存活”，而在于当前数据链路本身存在多处放大延迟的实现：

1. 总览页需要前端并发拼接 9 个接口。
2. `/api/budgets` 与 `/api/budgets/alerts` 对每一条预算都单独做一次交易聚合，属于典型 N+1。
3. 多个高频统计接口只按 `userId` 查询，没有按当前 `accountId` 收口。
4. `/api/metrics/consumption/summary` 没有实际使用前端传入的 `platform` 与 `search`。

在主账号已有约 `12735` 条交易数据的情况下，这些问题会叠加成明显卡顿，并让用户误以为后端不稳定。

---

## 2. 优化目标

1. 把总览页的多请求拼装改为单接口聚合。
2. 让高频统计接口只查询当前账户数据。
3. 去掉预算统计中的 N+1 聚合查询。
4. 保证总览筛选条件真的传到后端并生效。

---

## 3. 本次调整

### 3.1 后端接口

1. 新增 `/api/dashboard/summary`，由后端一次性返回总览页所需聚合数据。
2. 修复 `/api/metrics/consumption/summary`，补上 `accountId`、`platform`、`search` 条件。
3. 修复消费统计相关接口，统一改为按当前账户查询。
4. 修复 `/api/assets`、`/api/savings`、`/api/budgets`、`/api/budgets/alerts` 的账户范围。
5. 去掉消费总览接口里重复的 `console.time` 调试计时，避免并发请求时产生日志噪音。

### 3.2 预算统计

1. `budgets` 与 `budget alerts` 不再对每条预算单独跑一次数据库聚合。
2. 统一改为先取当前账户、当前统计窗口内的支出交易，再在内存中复用 `calculateBudgetHealth` 计算各条预算状态。

### 3.3 前端总览页

1. `web/src/features/dashboard/data-loader.ts` 改为只请求 `/api/dashboard/summary`。
2. 保留原有热缓存键设计，但把后端聚合职责移到服务端完成，减少浏览器往返和数据拼装。

---

## 4. 验证结果

1. `npm.cmd --prefix server run build` 通过。
2. `npm.cmd --prefix web run typecheck` 通过。
3. `npm.cmd run build` 通过。
4. 以本地真实开发数据压测，新的 `/api/dashboard/summary` 热态响应约为 `86ms`。

---

## 5. 结论

这次优化后，总览页的主要瓶颈从“前端拆分请求 + 后端重复聚合”改为“单次后端聚合 + 前端直接消费结果”。后续如果用户仍感知到“后端不稳定”，就应该优先排查本地是否存在重复启动的 Node 进程、数据库连接状态或首次冷启动编译，而不再是当前总览数据链路本身。
