# OpenStar 前端目录收口开发文档

## 0. 文档版本

- 大版本：`v1.0`
- 小版本：`v1.0.0`
- 本次更新：明确 `web/` 为唯一前端主入口，补充旧前端清理清单，并修正根目录遗留构建说明。

---

## 1. 背景

当前仓库同时存在两套前端目录：

- 生效前端：`web/src`
- 历史前端副本：`src` 下除 `server` 外的前端目录

同时还存在一套根目录级的旧前端配置：

- `Dockerfile`
- `README.md`
- `components.json`
- `next.config.ts`
- `tsconfig.json`
- `public/`

如果不先完成入口收口，后续继续开发时容易出现以下问题：

- 新代码误写到 `src/` 而不是 `web/src/`
- Docker 或脚手架工具误指向旧前端
- 后续删除旧目录时无法快速判断哪些文件已经迁移完成

---

## 2. 当前结论

### 2.1 生效中的目录

- 前端主目录：`web/`
- 后端主目录：`src/server/`

### 2.2 已核对的主入口链路

以下链路已经统一指向 `web/`：

- 根目录 `package.json`
- `playwright.config.ts`
- `.github/workflows/ci.yml`
- `.github/workflows/deploy-non-docker.yml`
- `docker-compose.yml`
- `src/server/scripts/deploy-linux.sh`

### 2.3 `src` 与 `web/src` 的重叠情况

- 同路径文件总数：`102`
- 内容完全一致：`75`
- 同路径但内容已分叉：`27`
- 仅存在于旧前端 `src/` 的前端文件：`1`

说明：

- `src` 下的前端并不是零散残留，而是一套高度接近 `web/src` 的历史副本。
- 由于已有 `27` 个同路径文件分叉，不能直接批量删除旧前端，必须先以 `web/src` 为主做最后一轮人工核对。

## 5. 本轮收口动作

本轮先做不会影响现有业务功能的基础收口，不直接删除旧前端源码：

1. 重写根目录 `README.md`，明确 `web/` 为唯一前端主入口。
2. 修正根目录 `Dockerfile`，从仓库根构建时直接打包 `web/` 前端，而不是依赖旧根目录入口。
3. 新增本开发文档，固化分叉文件和迁移顺序，避免后续重复盘点。

这样做的原因：

- 当前工作区存在未提交改动，直接删除旧前端容易和现有变更冲突。
- 先统一入口，再逐步清理源码，风险更低。

---

## 6. 后续清理顺序

### 阶段一：入口统一

- 保持 `web/` 为唯一前端入口。
- 禁止新增业务代码到根目录 `src/` 前端。
- 所有前端验证命令统一在 `web/` 或根目录包装脚本执行。

### 阶段二：分叉文件核对

- 以 `web/src` 为主版本。
- 对第 3 节中的 `27` 个分叉文件逐个核对，确认旧前端中不存在未迁移逻辑。
- 核对过程中如发现仍需保留的逻辑，先补进 `web/src`，再删除旧文件。

### 阶段三：旧前端删除

待第 3 节全部核对完成后，删除以下目录：

- `src/app`
- `src/components`
- `src/features`
- `src/lib`
- `src/themes`
- `src/types`
- `public/`

### 阶段四：旧配置删除

待旧前端源码删除完成后，再评估是否删除或改造以下根目录遗留配置：

- `components.json`
- `next.config.ts`
- `tsconfig.json`

---

## 7. 验收标准

完成完整收口后，应满足以下条件：

1. 仓库内只保留 `web/` 这一套前端源码。
2. 所有前端构建、测试、部署命令都不再依赖根目录旧前端目录。
3. `src/` 仅保留 `server/` 后端。
4. 新成员进入仓库后，不会再把前端代码误写到旧目录。
