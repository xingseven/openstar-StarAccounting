# 数据库结构核对报告

## 1. 核对范围

本次核对基于以下两部分：

1. 代码模型：`src/server/prisma/schema.prisma`
2. 实际数据库：`src/server/.env` 中配置的 MySQL 库 `Star_Accounting`

核对时间：2026-03-23

## 2. 核对结论

结论很直接：

- 当前数据库不缺表
- 当前数据库不缺字段
- 当前数据库不缺外键
- 当前数据库不缺索引
- 字段类型、可空、默认值与 Prisma 模型一致

也就是说，当前数据库结构已经可以支撑现有 Web 端和连接功能开发，不需要立刻补建新表。

## 3. 当前已存在的业务表

实际数据库与 Prisma 模型一致，包含以下 14 张表：

- `account`
- `account_member`
- `aimodelconfig`
- `appconnection`
- `asset`
- `budget`
- `exchangerate`
- `importerrorlog`
- `loan`
- `savingsgoal`
- `savingsplan`
- `themeconfig`
- `transaction`
- `user`

## 4. 当前连接功能是否缺表

当前答案是：不缺。

原因：

- Web 端生成验证码时，已经写入 `appconnection`
- App 端提交 OTP 时，已经基于 `appconnection` 做校验
- 设备授权成功后，设备信息也已经回写到 `appconnection`
- Web 端的“已授权设备列表”和“撤销授权”也已经依赖 `appconnection`

所以以你现在的开发阶段来看，`appconnection` 这一张表已经足够支撑：

1. 一次性验证码生成
2. App 首次绑定
3. 设备授权管理
4. 撤销设备访问

## 5. 发现的细节

本次核对没有发现结构性缺失，只发现两个“数据库里有、Prisma 没显式声明”的索引：

- `themeconfig.ThemeConfig_accountId_fkey`
- `user.User_defaultAccountId_idx`

这不影响运行。通常这类索引是数据库在外键关系上自然形成的辅助索引，属于“可接受差异”，不是故障。

## 6. 当前阶段的建议

建议现在不要急着改数据库。

原因：

- 你当前的连接流程已经能跑通
- 现在补很多“未来可能会用到的表”，大概率会提前设计过度
- 先把 App 的第一版绑定、登录、同步流程做出来，比先扩表更重要

当前最合理的策略是：

1. 保持现有表结构不变
2. 继续开发 App 首次连接流程
3. 等到出现真实业务需求后，再补对应表

## 7. 后续可能需要新增的表

虽然现在不缺表，但如果后面 App 功能继续扩展，下面这些表是“可能会需要”的。

### 7.1 设备刷新令牌表

建议表名：`device_refresh_token`

适用场景：

- 设备长期登录
- Access Token 过期后自动续签
- 一个设备允许多个会话

如果未来你不想一直用 `dev-connectionId` 这种轻量令牌，而是要做正式设备登录体系，这张表就会需要。

### 7.2 设备活跃心跳表

建议表名：`device_heartbeat`

适用场景：

- 在 Web 端显示“设备是否在线”
- 记录最后活跃时间
- 判断某个设备多久没同步了

如果你后面只需要“设备列表 + 撤销”，那暂时不用建；如果你想显示“手机在线中/最后活跃于 5 分钟前”，就需要。

### 7.3 数据同步日志表

建议表名：`device_sync_log`

适用场景：

- App 和服务端做交易同步
- 记录同步开始、结束、失败原因
- 追踪某次同步导入了多少条数据

如果后面要做“手机记账后自动同步到网页”，这张表会很有用。

### 7.4 推送令牌表

建议表名：`device_push_token`

适用场景：

- App 推送通知
- 预算提醒、还款提醒、同步结果通知

如果不做推送，可以完全不建。

### 7.5 连接尝试日志表

建议表名：`connection_attempt_log`

适用场景：

- 记录验证码输错次数
- 做限流或风控
- 排查非法设备尝试连接

这个表更偏安全审计，不是第一阶段必需。

## 8. 我建议的优先级

按你的当前阶段，我建议这样排：

### 现在就保留

- `appconnection`

### 下一阶段再看

- `device_sync_log`
- `device_heartbeat`

### 以后做正式设备体系时再加

- `device_refresh_token`
- `device_push_token`
- `connection_attempt_log`

## 9. 最终判断

如果你的目标只是：

- 网页生成验证码
- App 输入服务器地址和验证码
- 完成首次绑定
- 后续通过设备令牌访问接口

那么当前数据库已经够用，不缺表。

如果你的目标升级为：

- 多设备长期登录
- 自动同步
- 在线状态
- 推送通知
- 风控审计

那时才需要继续补表。

## 10. 下一步建议

我建议接下来优先做以下两件事，而不是继续改库：

1. 完成 App 侧“输入服务器地址 + OTP 绑定”的首版流程
2. 设计第一版“交易同步接口”，确认 App 到服务端的数据流

等你准备做“同步记录”或“长期设备登录”时，再回来补表，成本最低，也最稳。
