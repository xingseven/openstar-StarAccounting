# openstar-StarAccounting Flutter 统一前端迁移开发文档

## 1. 文档目的

这份文档用于回答一个明确的问题：

**如果我们决定以后以性能优先为目标，把当前项目重构为 Flutter Web + Flutter App 共用一套业务前端，应该怎么迁，迁哪些，不迁哪些，按什么顺序迁。**

本文档面向当前仓库真实情况编写，不是通用模板。

当前项目现状：

- Web 前端：`Next.js 16 + React 19 + TypeScript`
- 后端：`Node.js + Express + TypeScript + Prisma`
- 数据库：`MySQL`
- 当前移动端思路：Web/PWA 能力已具备，APK 更新链路有设计，但还没有原生 Android/iOS 工程

本文档的目标不是“立刻开工写 Flutter 代码”，而是先把迁移路线、模块边界、后端复用策略、风险和验收标准一次说清楚。

---

## 2. 迁移结论先说

如果目标是：

- Web 端比现在更流畅
- App 端长期性能更好
- Web 和 App 尽量共用一套业务前端

那么最合适的路线不是继续维护当前 `Next.js + 套壳 App`，而是：

1. **保留当前 Node.js 后端**
2. **保留当前 MySQL 数据库**
3. **将登录后的业务前端重构为 Flutter**
4. **Flutter 同时输出 Web、Android、iOS**
5. **当前 Next.js Web 在迁移期间继续在线，直到 Flutter Web 可替换**

一句话总结：

**后端基本不换，核心是前端重构。**

---

## 3. 为什么要迁

当前项目已经具备完整业务系统雏形，但随着页面、图表和交互逐渐增多，继续沿着当前 Web 前端演进，会遇到下面这些问题：

### 3.1 当前项目的真实特点

- 页面多：总览、资产、消费、储蓄、贷款、预算、数据、AI、连接、设置、关于、主题、后台
- 图表多：折线图、柱状图、环形图、桑基图、热区矩阵、日历图、散点图等
- 数据密集：卡片、列表、筛选、弹层、图表在同一页面共存
- 交互多：分页、搜索、筛选、导入、上传、主题切换、设备绑定、AI 分析
- 后期还会继续加：同步、缓存、离线、系统通知、设备能力、安装更新

### 3.2 当前前端的问题

- Web 页面切换和状态切换已经出现“不够顺”的体感
- 多图表页面会持续吃渲染和重排成本
- 后续如果继续做真正移动端 App，WebView/套壳方案的上限偏低
- 想同时保证“高性能”和“Web/App 共用一套业务前端”，当前 React Web 路线不够理想

### 3.3 为什么是 Flutter

Flutter 更适合这个项目的原因是：

- 能同时覆盖 `Web + Android + iOS`
- UI 渲染和动画控制更统一，复杂交互更容易做顺
- 更适合做登录后的业务系统和高交互面板
- 可以把大部分业务页面、状态管理、接口调用、数据模型统一到一套代码
- 后端不需要跟着重写

## 4. 迁移后的目标架构

目标架构如下：

```text
                       +----------------------+
                       |   Flutter Frontend   |
                       |----------------------|
                       |  Web / Android / iOS |
                       +----------+-----------+
                                  |
                                  | HTTP / HTTPS
                                  v
                    +------------------------------+
                    |  Node.js + Express API       |
                    |  TypeScript + Prisma         |
                    +---------------+--------------+
                                    |
                                    v
                              +-----------+
                              |  MySQL    |
                              +-----------+
```

迁移后建议的职责划分：

- Flutter：负责所有登录后的业务界面、交互、状态、缓存、上传下载、图表展示
- Node.js 后端：继续负责认证、权限、业务计算、AI 接口整合、数据库写入、版本检查
- MySQL：继续负责业务数据存储

---

## 5. 不迁什么

这次迁移不是全项目推倒重来。下面这些内容原则上不作为第一阶段目标：

- 不重写后端为 Java、Go、Python 或其他语言
- 不更换 MySQL
- 不先做桌面端
- 不先做复杂离线双向同步系统
- 不强行把所有页面一次性全部迁完再上线
- 不尝试把现有 React 页面逐文件自动翻译成 Flutter 页面

特别说明：

**不要追求“现有 Web 代码直接复用到 Flutter”。**

能复用的是：

- 业务规则
- 接口契约
- 数据结构
- 页面信息架构
- 文案与交互流程

不能直接复用的是：

- React 组件代码
- Next.js 路由代码
- 大部分 Web UI 实现代码

---

## 6. 推荐总体策略

推荐采用：

**“后端不动主干，前端双轨迁移，最终切换”**

分成三条线并行推进：

### 6.1 线一：后端契约稳定

目标：

- 保证 Flutter Web 和 Flutter App 都能复用当前 API
- 在迁移开始前先把认证、分页、筛选、上传下载等接口规则收紧

### 6.2 线二：Flutter 新前端建设

目标：

- 新建 Flutter 业务前端工程
- 优先做登录后的核心业务模块
- 同时兼容 Web 和移动端布局

### 6.3 线三：旧 Web 持续在线

目标：

- 当前 Next.js 版本继续作为正式可用版本
- Flutter 版本达到可用线后，再灰度切换

当前仓库建议不要立刻大改目录结构，先在现有仓库中增加 Flutter 前端目录。

推荐结构：

```text

├── src/
│   └── server/                    # 继续保留 Node.js 后端
├── web/                           # 旧 Next.js 前端，迁移期继续保留

│   │   ├── bootstrap/

│   │   ├── shared/
│   │   ├── features/
│   │   ├── routing/
│   │   └── main.dart
│   ├── test/
│   ├── integration_test/
```

- 便于同一个 Git 仓库里维护文档、接口和迁移进度

## 8. Flutter 技术选型建议

下面是面向当前项目的推荐选型，不是唯一方案，但这是平衡性能、可维护性和开发效率后的建议。

### 8.1 基础技术

- Flutter Stable
- Dart Stable

### 8.2 路由

- `go_router`

原因：

- 支持 Web 路由和移动端路由统一
- 支持嵌套路由、鉴权守卫、重定向
- 适合后台管理类和多模块系统

### 8.3 状态管理

- `flutter_riverpod`

原因：

- 比较适合中大型业务系统
- 全局状态、异步状态、依赖注入都能统一处理
- 对测试和模块拆分更友好

### 8.4 网络层

- `dio`

建议搭配：

- 统一 `ApiClient`
- 统一拦截器
- 统一错误映射
- 统一 Token 注入

### 8.5 数据模型

- `freezed`
- `json_serializable`

原因：

- 适合将后端返回数据映射为稳定的 Dart 模型
- 便于维护复杂业务对象

### 8.6 本地缓存与离线数据

推荐优先级：

1. 轻量缓存：`shared_preferences`
2. 安全凭证：`flutter_secure_storage`
3. 结构化缓存：`drift` 或 `isar`

当前项目建议：

- 第一阶段先做 `Token + 用户信息 + 主题 + 最近筛选条件` 的本地持久化
- 第二阶段再补交易列表、资产列表等结构化缓存

### 8.7 图表方案

当前项目图表复杂度较高，建议采用分层方案：

- 常规图表：折线图、柱状图、饼图、环形图
- 复杂图表：日历热图、热区矩阵、桑基图、特殊统计面板

推荐策略：

- 先用成熟 Flutter 图表库承接常规图表
- 对高度定制的图表，逐步转为 `CustomPainter`

建议原则：

- 不要一开始就追求 100% 还原当前所有图表形态
- 优先保证性能、信息结构和交互节奏
- 对最复杂的图表，允许第一版先降级展示

### 8.8 动画策略

推荐：

- 页面切换：统一路由过渡
- 面板弹出：统一底部抽屉和对话框动效
- 数据切换：统一骨架屏、淡入淡出、列表局部更新

原则：

- 动画要克制，优先提升“顺滑感”，不是为了炫技
- 数据重刷时避免整页闪烁

## 11. 后端保留策略

### 11.1 可以保留什么

当前后端原则上可以继续保留：

- `Express` 路由层
- `Prisma` 数据访问层
- `MySQL` 模型结构
- 用户认证逻辑
- 预算、资产、交易、贷款、储蓄等业务逻辑
- AI 对接逻辑
- 连接码与设备授权逻辑
- 更新清单接口

### 11.2 后端需要补哪些整理

虽然不用换后端，但建议补一轮“接口契约整理”。

至少应处理这些点：

1. 统一返回结构  
   当前建议统一保持：
   ```json
   {
     "code": 200,
     "message": "ok",
     "data": {}
   }
   ```

2. 统一错误码  
   Flutter Web 和 App 需要稳定错误码，不能只靠中文错误文案判断。

3. 统一分页结构  
   列表接口建议统一：
   ```json
   {
     "items": [],
     "page": 1,
     "pageSize": 20,
     "total": 100
   }
   ```

4. 统一筛选参数  
   日期范围、平台、分类、关键词、排序方式要统一命名。

5. 统一认证头  
   Web 与 App 都统一走 Bearer Token。

6. 补刷新机制  
   当前如果只有单一 Token，建议补 `refresh token` 方案，避免 Flutter App 长期登录体验差。

7. 统一上传下载协议  
   AI 扫描图片、账单导入、APK 下载、导出文件都要明确协议。

8. 处理 CORS 和缓存头  
   Flutter Web 会直接访问 API，需要把跨域策略明确。

### 11.3 后端不建议现在做的事

- 不建议迁移阶段同时引入微服务
- 不建议为了 Flutter 再额外套一层新 BFF
- 不建议先把数据库模型大改一轮再开始前端重构

原则：

**先保证现有接口能被 Flutter 稳定消费，再考虑高级重构。**

## 12. 设计系统与交互策略

如果希望 Web 和 App 共用一套前端，不只是代码共用，更关键的是：

- 信息架构统一
- 交互节奏统一
- 设计 token 统一

### 12.1 设计 token 统一

在 Flutter 里应先定义：

- 颜色系统
- 字体层级
- 间距系统
- 圆角系统
- 阴影系统
- 动效时长
- 图表色板

建议不要直接逐页“照抄 CSS”，而是先抽象出 token。

### 12.2 Web 与 App 的布局差异

虽然是同一套 Flutter 代码，但布局不能完全一把梭。

建议这样处理：

- Web：侧边栏 + 顶栏 + 主工作区
- 平板：紧凑侧栏或折叠导航
- 手机：底部导航 + 页面内分段布局

原则：

- 共用模块，不强求共用一模一样的页面骨架
- 让“同一个业务组件”在不同端用不同壳子承载

### 12.3 动效与切换标准

统一标准如下：

- 页面切换不超过 300ms
- 弹层动画不超过 240ms
- 筛选切换尽量只刷新局部区域
- 图表刷新默认采用渐变更新，不整页重建

## 13. 性能专项要求

既然本次迁移的核心目标是性能，就必须在文档里先定下性能红线。

### 13.1 目标体验

- 常规页面滚动稳定
- 主要交互无明显卡顿
- 图表切换不出现长时间白屏
- 页面返回不重复全量重建
- 中端 Android 设备上能保持可接受流畅度

### 13.2 Flutter 侧必须执行的优化

1. 按模块懒加载
2. 大列表分页加载
3. 图表按需渲染，不在首屏一次性挂满
4. 避免大面积 `setState`，用更细粒度状态更新
5. 重计算任务尽量放 isolate 或后台处理
6. 页面保活策略要明确，不要切一次页就整页重拉数据
7. 网络请求要做取消、防抖和去重

### 13.3 Web 侧额外注意

Flutter Web 不会自动在所有场景都比 Next.js 更快，所以要特别控制：

- 首屏资源量
- 首次字体和图标加载
- 图表初次创建成本
- 大列表和复杂面板的首屏堆叠量

建议：

- 首屏只放关键概览
- 非核心图表延迟挂载
- 把“高级分析区”拆到二级区域或标签页

- 确认 Flutter 目录结构和基础依赖

任务：

- [ ] 梳理全部业务接口清单

- [ ] 冻结数据库关键模型字段命名
- [ ] 确定 Flutter 项目目录
- [ ] 确定设计 token 基线

交付物：

- API 契约清单
- Flutter 工程骨架
- 迁移排期表

建议工期：

- 3 到 5 天

## 阶段 1：后端契约整理

目标：

- 在不换后端的前提下，让 Flutter 端能稳定接入

任务：

- [ ] 统一认证响应

## 阶段 2：Flutter 基础框架搭建

目标：

- 能跑通 Web 和 Android 双端开发环境

- 数据模型
- 图表组件封装

需要按平台分支处理的内容包括：

- 系统通知
- 安装更新

## 18. 风险与规避

### 风险 1：误以为 Flutter Web 一定全方位比 Next.js 快

事实：

- Flutter Web 的业务一致性更强
- 复杂交互控制更强
- 但首屏、SEO、浏览器生态并不天然全面占优

规避：

- 只把“登录后的业务系统”统一到 Flutter
- 如果以后有官网、文档站、SEO 页面，继续交给 Next.js

### 风险 2：图表迁移难度被低估

事实：

- 当前项目图表种类很多
- 桑基图、热区矩阵、日历图不一定能一把迁完

规避：

- 先保证数据正确和阅读路径正确
- 最复杂图表先允许降级
- 后续逐步换成高性能自绘方案

### 风险 3：后端契约不稳定导致前端反复返工

规避：

- 迁移开始前先做接口整理
- 不要边做 Flutter 边随意改接口字段

### 风险 4：一次性全迁导致周期过长

规避：

- 保留旧 Web
- 分模块迁
- 先迁最核心模块

### 风险 5：Web 和 App 强行共用同一种布局

规避：

- 共用业务模块，不强行共用完全一致的页面骨架

## 24. 开始迁移前的注意事项

下面这些注意事项很重要，建议在真正开工前先统一认识。

### 24.1 不要一上来先迁消费页

消费分析是当前项目里最复杂、最重、最容易卡住的模块之一：

- 图表种类最多
- 筛选最多
- 状态切换最多
- 联动逻辑最多

如果第一刀就先迁消费页，很容易在早期阶段被复杂图表和布局细节拖慢。

正确顺序应该是：

1. 登录
2. 壳层
3. Dashboard
4. 资产
5. 数据管理
6. 最后再迁消费分析

### 24.2 不要边迁边频繁改后端字段名

Flutter 前端重构最怕的不是写页面，而是接口契约不断变化。

如果后端字段今天叫 `createdAt`，明天又改成 `createTime`，前端模型、解析、列表、筛选、排序都会跟着返工。

建议：

- 开始迁移前先冻结一轮接口字段
- 需要调整时优先兼容旧字段
- 统一在接口契约文档里登记变更

### 24.3 不要追求把 React 页面逐个“翻译”成 Flutter

这条非常关键。

Flutter 不是 React 的另一种语法皮肤，不能按“页面对页面、组件对组件、CSS 对 Widget”机械翻译。

正确做法是：

- 复用业务结构
- 复用接口逻辑
- 复用信息架构
- 重新设计 Flutter 里的组件和布局层次

如果强行按旧页面一比一翻译，结果通常是：

- 代码难维护
- 性能没有真正提升
- Flutter 写法变得很别扭

### 24.4 不要一开始就追求 100% 视觉复刻

迁移的第一目标是：

- 更流畅
- 更稳定
- 更易维护

不是：

- 第一版就把所有卡片阴影、间距、动效、图表细节复刻到像素级一致

建议：

- 第一版先保证信息层级正确
- 第二版再打磨细节视觉
- 第三版再补高级动画和复杂图表

### 24.5 Flutter Web 和 Flutter App 不能完全按同一布局硬套

虽然是同一套前端代码，但 Web 和移动端的最佳布局不一样。

建议：

- Web 用侧边栏工作区布局
- 手机用底部导航和分段内容布局
- 共用模块，不强求共用完全相同的页面骨架

错误做法是：

- 把 PC 页面直接缩到手机
- 或把手机布局硬套到大屏 Web

### 24.6 不要忽略首屏性能

很多人迁到 Flutter 后，会只关注滚动和页面切换，却忽略 Web 首屏加载。

你这个项目图表很多，如果首屏一次性挂载太多内容，即使 Flutter 版也可能不轻。

建议：

- 首屏只展示核心指标和关键模块
- 非关键图表延迟渲染
- 高级分析区放到二级区域或标签页

### 24.7 图表迁移要先分级，不要一口气全做

建议把图表分 3 级：

1. 常规图表  
   折线图、柱状图、饼图、环形图，优先迁

2. 中等复杂图表  
   日历图、热区矩阵、组合图，第二批迁

3. 高复杂图表  
   桑基图、特殊统计图、自定义联动画布，最后迁

如果不分级，很容易在前两周就陷进最难图表，导致主流程迟迟跑不起来。

### 24.8 认证方案要提前定，不要后补

Flutter Web 和 App 共用时，认证会比纯 Web 更敏感。

建议提前确定：

- Access Token 存哪里
- Refresh Token 是否启用
- Token 失效后怎么自动续期
- Web 和 App 是否使用同一套认证头
- 设备授权接口是否和普通登录分开

如果这些问题拖到中后期再补，登录态会反复返工。

### 24.9 文件上传下载要提前验证

你这个项目不只是普通表单系统，还涉及：

- CSV 导入
- 图片上传
- 更新包下载
- 导出文件

这些能力在 Flutter Web 和 App 上的表现不完全一样。

建议在最早阶段就验证：

- Flutter Web 上传 CSV 是否稳定
- Flutter Web 上传图片是否稳定
- Android 下载文件保存到哪里
- Web 下载和 App 下载是否共用同一个接口

### 24.10 不要在迁移前同时做大规模数据库重构

迁移前端已经是大工程，如果同时改数据库模型、字段命名、权限模型、同步机制，风险会成倍增加。

建议：

- 前端迁移期尽量保持数据库主结构稳定
- 只做必要的接口修正
- 数据库大改放到迁移稳定之后

### 24.11 旧 Web 不要过早下线

即使 Flutter Web 做出来了，也不要第一天就把旧 Web 删除。

建议至少保留：

- 一段灰度期
- 一个回滚入口
- 一份功能对照清单

推荐做法：

- 旧 Web 继续在线
- Flutter Web 挂测试地址或新入口
- 核心业务验收后再切主入口

### 24.12 迁移期间要控制需求膨胀

最常见的问题不是技术做不动，而是迁移时不断顺手加需求。

比如：

- 顺手换设计语言
- 顺手加新首页
- 顺手重做后台权限
- 顺手重构 AI 模块

这样很容易把“迁移”变成“重做一整个新产品”。

建议：

- 迁移期只做必要升级
- 新需求单独排期
- 先保功能闭环，再加扩展功能

### 24.13 一定要设验收基线

如果没有明确基线，最后很容易陷入“感觉差不多了”但又谁都不敢切换。

建议在开始前先写清楚：

- 哪几个页面算第一批必须可用
- 哪些图表允许暂时降级
- 哪些接口必须完全兼容
- 性能达到什么程度算过线
- 什么条件下允许替换旧 Web

### 24.14 最后一个原则：先跑通，再做好看

整个迁移期最应该坚持的一条原则是：

**先把系统跑通，再把体验打磨到优秀。**

先后顺序建议始终保持：

1. 路由和登录跑通
2. 数据获取跑通
3. 核心 CRUD 跑通
4. 核心页面稳定
5. 复杂图表补齐
6. 动效和视觉精修

这样才能保证项目一直在向可交付版本推进，而不是一直停留在“局部很好看，但整体还不能用”。
