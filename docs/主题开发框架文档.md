# OpenStar 主题开发框架指南 (v2.1)

本文档说明当前仓库的主题开发框架。当前真正生效的主题代码位于 `web/src`，并且从本次整理开始由主仓库统一追踪，不再单独作为嵌套 Git 仓库维护。后续如果迁移到 Flutter，可沿用同样的分层思路。

当前仓库的源码边界已经收口为两棵主目录：
- `web/src`：当前正式生效的前端、主题与 Dashboard 实现
- `server/src`：当前正式生效的后端源码

根目录旧的 `src/` 不再承载正式前后端代码；如果本地还看到 `src/server`，那是整理前遗留的空目录，应直接清理，不能再继续往里面放文件。

## 0. 文档版本

- 大版本：`v2.1`
- 小版本：`v2.1.25`
- 本次更新：补充“已经接入公共骨架框架的主题页，如果骨架与真实模块的尺寸、高度、表格密度仍不一致，允许在页面级 LoadingShell 继续保留更细粒度的专用镜像块，不强迫全部退回通用占位”的约定，并记录默认总览主题骨架继续收口的处理方式。

## 1. 当前主题架构

当前主题系统不是“所有主题都往一个页面里加 `themeId` 判断”，而是拆成三层：

### 1. Token 层

文件：`web/src/themes/registry.ts`

职责：
- 定义 `ThemeId`
- 维护主题名称、描述、预览信息
- 维护整套 CSS 变量（颜色、阴影、圆角语义、输入框、空状态等）

这一层只负责“主题长什么样”，不负责“主题在应用里怎么接线”。

### 2. Manifest 层

文件：`web/src/themes/theme-manifest.ts`

职责：
- 定义某个主题使用哪套 Dashboard 变体
- 定义 Header 是否透明
- 定义移动端底部导航变体
- 定义 Sidebar 使用哪种壳层布局
- 定义主题选择页的预览 tone

这一层只负责“主题接哪套壳层和页面变体”，不放颜色 token。

### 3. Registry / Loader 层

文件：`web/src/themes/dashboard-registry.tsx`

职责：
- 集中注册 Dashboard 变体与动态加载逻辑
- 统一维护 LoadingShell
- 统一维护“当前主题实际对应哪个 Dashboard 入口文件”

这一层解决的是：后续新增几十个主题时，`page.tsx` 不需要继续写长串条件分支。

补充约定：
- LoadingShell 不是独立装饰层，必须和真实 Dashboard 的外层容器、背景壳层、栅格结构、模块顺序同步演进。
- 如果某个 Dashboard 变体删除了旧背景壳、调整了容器间距，或者重排了模块顺序，匹配的 LoadingShell 必须在同一次改动里同步更新。
- 如果某个功能页被明确要求“按已有主题文件来开发”，优先复用该主题的卡片层级、强调色角色和信息组织方式，不要混入另一套通用 Hero 结构。
- 如果需求里只强调“主题一致”“风格一致”或“按某个主题来改”，默认对齐的是卡片语言、色彩关系、圆角阴影、节奏和页面壳层，不默认等于整页模块骨架也必须逐块复制；功能页仍应优先围绕自身业务语义重排模块。
- 只有当用户明确要求连“布局 / 骨架 / 顶部卡片排法”也一起一致，或者明确点名要“直接照着目标主题结构做”，才直接复用目标主题的行列骨架、卡片组合和对应 LoadingShell 结构，允许只替换业务文案与数据映射。
- 如果原功能页已经有更完整的分析模块或更多图表，主题对齐默认只调整视觉语言与模块编排，不应为了“套主题”主动删减关键图表；只有用户明确要求裁剪范围时才允许减少模块。

## 3. 页面分层原则

### Dashboard 总览页

Dashboard 允许存在多个独立文件，因为不同主题的首页布局差异最大，视觉特征也最强。

但现在不再直接在 `page.tsx` 里写：

```tsx
themeId === "analytics" ? ...
```

而是统一走：
- `theme-manifest.ts` 决定该主题属于哪种 Dashboard 变体
- `dashboard-registry.tsx` 决定该变体实际加载哪个组件

如果要快速反查“当前主题到底落到哪个总览文件”，现在可以直接按下面这条链路找：

- `web/src/app/(dashboard)/page.tsx`
- `getDashboardRoutePath(themeId)`
- `web/src/app/(dashboard)/[dashboardEntry]/page.tsx`
- `getDashboardThemeComponent(themeId)`
- `web/src/themes/dashboard-registry.tsx` 里的 `DASHBOARD_ENTRY_FILES`
- `web/src/features/dashboard/components/themes/[ThemeName]Dashboard.tsx`

现在总览页同时保留两个入口：

- `/`：稳定根入口，直接渲染当前主题对应的总览页面。
- `/<Dashboard文件名>`：按主题区分的别名路由，例如 `http://localhost:3000/DefaultDashboard`。

当用户直接访问到“与当前主题不一致”的 Dashboard 文件名别名时，`DashboardPageShell.tsx` 会自动纠正到当前主题对应的路由；如果当前已经在根路径 `/`，则保持原地渲染，避免与侧边栏点击发生路由竞争。

目前注册表里的总览文件名映射如下：

- `default` -> `DefaultDashboard.tsx`
- `analytics` -> `AnalyticsDashboard.tsx`
- `orange-purple` -> `OrangePurpleDashboard.tsx`
- `dusty-blue` -> `DustyBlueDashboard.tsx`
- `vibrant` -> `VibrantDashboard.tsx`
- `charming-purple` -> `CharmingPurpleDashboard.tsx`
- `white-grid` -> `WhiteGridDashboard.tsx`

其中 `graphite`、`spruce`、`terracotta`、`nova`、`midnight`、`frost` 当前仍通过 manifest 复用 `default`，所以实际也会落到 `DefaultDashboard.tsx`。

### 其他功能页

其他功能页仍然优先共用一套业务页面，但必须遵守下面的规则：

1. 只通过 CSS 变量和共享主题原语换皮。
2. 不允许在业务页面里堆积大量 `themeId === "xxx"` 判断。
3. 如果只是颜色、阴影、圆角、字体节奏不同，不拆文件。
4. 如果模块顺序、信息密度、卡片结构开始明显不同，先抽 `feature config`。
5. 如果某个主题在某个功能页已经出现明显独立布局，再拆独立 override 或独立页面文件。

### 共享悬浮筛选器

文件：`web/src/components/shared/FloatingFilter.tsx`

当前约定：

- 消费页、总览页以及后续主题需要悬浮筛选入口时，优先复用这个共享组件。
- 如果页面已经有自己的筛选状态，就走受控模式：传入 `dateFilter`、`customPeriod`、`platform`、`searchQuery` 以及对应回调。
- 如果页面暂时只需要可点击的 UI，不准备立刻把数据查询接上，也可以只传 `isOpen`、`onOpenChange` 和 `transactionCount`，组件会回退到内部状态，避免出现“弹层能开，但里面按钮点不动”。
- 一旦页面需要让筛选结果驱动真实数据，请再把页面级状态与查询逻辑接回父组件，不要把业务请求直接塞回共享组件里。
- 总览页当前的正确接法是：`DashboardPageShell.tsx` 维护筛选状态，`features/dashboard/data-loader.ts` 按 `startDate/endDate/compareStartDate/compareEndDate/platform/search` 查询并缓存，再把状态与回调继续下传到 `DefaultDashboard.tsx` 和 `FloatingFilter.tsx`。
- 如果总览页内部还要维护 `isOpen` 这类纯 UI 浮层开关，必须把它隔离在轻量外层或单独子树，不要让该状态直接挂在承载 Recharts 图表的大组件上，避免开关筛选浮层时整页图表重复渲染。
- 如果消费页、资产页、总览页这类重图表页面还要支持本地平台 / 关键词筛选，筛选输入状态与重聚合 / 重图表结果状态也必须分层；输入优先留在轻量外层，承载图表的大子树通过 `useDeferredValue` 或同等低优先级机制消费延迟后的筛选值，避免输入过程中逐字触发整页重算。
- 移动端 `BottomSheet` 里的自定义年月选择也应优先复用 `YearStepper` / `MonthStepper`，保持与桌面端一致的左右步进交互，不再额外弹出月份下拉选择器。
- 其中 `全部时间` 必须显式允许 `startDate/endDate` 为空，数据加载器只能在“完全未传 query”时使用默认本月，不能把用户主动选出的空区间覆盖掉。
- `FloatingFilter.tsx`、`BottomSheet`、`Select` 这类共享筛选/弹层控件的主题跟随，优先通过 `web/src/components/shared/theme-primitives.tsx` 暴露的共享主题原语类名，以及 `var(--theme-...)`、模块 accent token 完成，不在共享组件里硬编码默认灰白样式。
- 如果差异只落在背景、边框、阴影、文字色、激活态或输入框表面样式，不允许在共享组件里继续堆 `themeId === "xxx"` 分支，也不要为单个主题复制一份筛选器或底部弹层。
- 只有当布局结构、交互流程或信息层级明显不同，才允许新增变体或页面级 override；纯样式差异继续留在共享组件中由 token 吸收。默认总览页可以依赖 `FloatingFilter` 的默认 `dashboard` 模块 accent，其他业务页则显式传入 `module="consumption"`、`module="assets"` 等模块语义。

### 共享加载骨架框架

文件：`web/src/components/shared/PageLoadingShell.tsx`

当前约定：

- 页面级 LoadingShell 不再优先各自手写整页骨架，优先复用 `LoadingPageShell`、`LoadingHeroShell`、`LoadingCompactHeroShell`、`LoadingMetricGrid`、`LoadingSplitSurface`、`LoadingChartSurface`、`LoadingListSurface`、`LoadingTableSurface`、`LoadingCardGrid` 等共享块。
- 允许页面保留少量和真实布局强绑定的局部骨架片段，但外层容器、通用卡片、表格、图表、列表、hero、指标卡应尽量回收到共享框架。
- 新页面新增加载态时，先判断能否直接由共享骨架块组合完成；只有在布局明显异于现有框架时，才补新的共享块或保留页面局部实现。
- 页面真实布局调整后，应先修改共享骨架块或基于共享骨架块的组合层，避免多个 LoadingShell 各自漂移。
- 如果页面已经挂上共享骨架框架，但用户仍能明显感知到“模块大小、高度、表格密度、标题区留白”与真实页面不一致，应优先在页面级 `LoadingShell` 继续细化镜像块，而不是为了复用强行维持通用骨架。

主题接入链路：

- 主题不会在 `theme-manifest.ts` 里直接声明“使用哪套骨架框架”，Manifest 只负责声明页面变体。
- 实际接线顺序是：`themeId -> theme-manifest.ts -> dashboardVariant -> dashboard-registry.tsx -> XxxLoadingShell -> PageLoadingShell 共享骨架块`。
- 也就是说，主题先选中页面变体；页面变体再决定加载时渲染哪个 `LoadingShell`；具体 `LoadingShell` 内部再去引用 `PageLoadingShell.tsx` 提供的公共骨架块。

后续新增主题时怎么接：

- 如果新主题只是换色、不换布局：直接在 `theme-manifest.ts` 里复用现有 `dashboardVariant`，不需要额外新建骨架框架；它会自动复用对应变体已有的 `LoadingShell`。
- 如果新主题需要独立 Dashboard 变体：新建 `XxxDashboard.tsx` 和 `XxxLoadingShell.tsx`，但 `XxxLoadingShell.tsx` 应优先使用 `PageLoadingShell.tsx` 的共享骨架块组装，不要重新手写整页骨架。
- 新的 `XxxLoadingShell.tsx` 建好后，要在 `dashboard-registry.tsx` 的 `loading` 配置中注册，再由 `theme-manifest.ts` 把主题指向该 `dashboardVariant`。
- 非 Dashboard 功能页也是同样原则：页面自身的 `LoadingShell` 去引用 `PageLoadingShell.tsx`，而不是把骨架框架信息写进 Manifest。

公共骨架迁移知识点：

- “迁移到公共骨架”不是把所有页面都塞进一个完全相同的骨架模板，而是把外层容器、通用卡片、图表占位、表格占位、列表占位、Hero/指标卡占位回收到 `PageLoadingShell.tsx`。
- 页面自己的 `LoadingShell` 应该变成“薄封装层”：只负责按真实页面的模块顺序组合共享骨架块，最多保留少量和真实布局强绑定的局部占位。
- 判断是否值得抽成共享块的标准，不是看它有没有在两个文件里重复，而是看它是否代表“稳定布局语义”，例如默认主题工作台卡片、默认主题强调指标卡、默认主题图表卡、默认主题表格卡。
- 迁移时优先对齐“真实页面结构”，而不是先追求骨架组件数量最少。宁可共享块多一层语义封装，也不要让骨架为了复用而和真实布局脱节。
- 默认主题页族当前可以分成两类：
  - 已经接入公共骨架：`ConsumptionLoadingShell.tsx`、`SavingsLoadingShell.tsx`、`LoansLoadingShell.tsx`
  - 仍需继续收口的默认主题页：`DashboardLoadingShell.tsx`、`AssetsLoadingShell.tsx`

公共骨架迁移注意点：

- 第一原则是模块顺序一致。真实页面先出现哪一行、哪一列、哪种主次关系，骨架也必须照着这个顺序渲染，不能为了省事把图表卡和列表卡交换位置。
- 第二原则是容器语义一致。真实页面如果已经移除了整页背景壳、大圆角外框或额外包裹层，骨架也必须同步删掉，避免再出现“骨架有一层大背景、真实页面没有”的错位。
- 第三原则是交互入口一致。像筛选器、表格头、管理面板这类有明确操作入口的区域，骨架要保留相同的头部关系和按钮占位，避免加载完成前后视觉重心跳变。
- 第四原则是不要把业务状态塞回共享骨架。共享骨架只表达布局，不感知真实数据、筛选条件、预算状态或资产分类逻辑。
- 第五原则是新增共享块后，优先让同一视觉语言下的多个页面共同复用；如果一个块只服务单页且没有稳定复用价值，就保留在页面局部 `LoadingShell`。
- 第六原则是迁移完要回头检查“默认主题页族是否已经闭环”。如果消费、储蓄、借贷已经接入，而总览页、资产页还保留整页手写骨架，就说明默认主题页族还没有真正收口完成。
- 第七原则是“通用框架优先，真实镜像兜底”。像默认总览页这种已经接入共享骨架的页面，如果首行大数字卡、预算卡、排行卡、表格卡仍和真实页面尺寸不一致，可以继续在该页 `LoadingShell` 内保留专用骨架细节，直到视觉跳变消失。

## 4. 新增主题的标准流程

假设新增主题 `neon`。

### 第一步：注册 Token

在 `web/src/themes/registry.ts` 中：

1. 在 `ThemeId` 中加入 `"neon"`
2. 在 `THEMES` 中补齐：
   - `name`
   - `description`
   - `preview`
   - `vars`

这一层只定义视觉 token，不决定页面接线。

### 第二步：注册 Manifest

在 `web/src/themes/theme-manifest.ts` 中新增 `neon`：

需要明确 5 件事：
- `dashboardVariant`
- `headerVariant`
- `mobileNavVariant`
- `sidebar.variant`
- `previewTone`

如果 `neon` 只是一个“换色主题”，通常直接复用：
- `dashboardVariant: "default"`
- `sidebar.variant: "default"`

### 第三步：判断是否需要独立 Dashboard

判断标准：

- 只换色、不换布局：不需要独立 Dashboard
- Hero、图表布局、模块顺序明显不同：需要独立 Dashboard

如果需要独立 Dashboard：

1. 新建 `web/src/features/dashboard/components/themes/NeonDashboard.tsx`
2. 新建 `web/src/features/dashboard/components/themes/NeonLoadingShell.tsx`
3. 在 `web/src/themes/dashboard-registry.tsx` 中注册新变体
4. 在 `theme-manifest.ts` 中把 `dashboardVariant` 指向这个新变体

其中 `NeonLoadingShell.tsx` 的推荐写法是：优先用 `PageLoadingShell.tsx` 里的共享骨架块做“薄封装”，只保留和 `NeonDashboard.tsx` 真实布局强绑定的少量局部占位，不要从零复制一整套骨架结构。

### 第四步：判断是否需要独立壳层

如果 `neon` 只是主题色不同，不要新增 Sidebar/Header 文件。

只有满足以下情况才新增壳层变体：
- 侧边栏宽度完全不同
- 导航结构完全不同
- 头部透明度或布局完全不同
- 主题选择页需要完全不同的预览 tone

### 第五步：验收与文档同步

新增主题完成后必须同步：
- `docs/主题开发框架文档.md`
- `docs/开发进度.md`
- `CHANGELOG.md`
- 如涉及发布版本，还要同步 `web/public/updates/latest.json`

## 5. 什么时候该拆文件

这是当前框架最重要的边界。

### 可以继续共用一个文件

满足以下情况时，共用页面即可：
- 只是换颜色
- 只是换阴影、圆角、边框
- 只是图表配色不同
- 只是局部文案 tone 不同

### 应该先抽配置

满足以下情况时，先抽 `feature config`，不要直接堆 `if`：
- 模块顺序有差异
- 指标卡密度不同
- Hero 内容组合不同
- 空状态样式不同

### 必须拆独立实现

满足以下情况时，不要再硬塞共享页：
- 页面结构已经明显不同
- 同一页面出现 3 个以上主题特判
- 布局、交互、模块层级已经不是同一套骨架

---

## 6. 当前推荐开发规则

后续所有主题开发默认遵守以下规则：

1. `registry.ts` 只放 token，不放页面分支逻辑。
2. `theme-manifest.ts` 只放接线规则，不放视觉 token。
3. `dashboard-registry.tsx` 只放 Dashboard 注册与动态加载，不写业务逻辑。
4. 共享业务页面里禁止继续增加主题名条件判断。
5. 共享页面优先吃 `var(--theme-...)` 和共享主题原语。
6. 一旦出现大量主题特判，立刻抽配置或拆独立变体。
7. 共享弹层、筛选器、下拉选择器优先通过 `theme-primitives.tsx` 和主题 token 跟随主题；结构不变时不拆主题分支。
8. Dashboard 自定义布局优先采用栅格拖拽，不把页面背景层直接改成自由画布；布局引擎只管理 widget 的 `x/y/w/h`，主题壳和卡片表面继续由现有主题系统负责。

补充文档：

- `docs/Dashboard拖拽布局开发文档.md`：用于后续实现总览页图表模块拖拽、缩放、布局持久化时的专项设计依据。

## 8. 反模式清单

以下做法禁止继续扩散：

- 在 `page.tsx`、`Header.tsx`、`Sidebar.tsx` 里继续增加长串 `themeId === "xxx"`
- 每新增一个主题就复制所有功能页
- 在业务组件里写死颜色而不走 `var(--theme-...)`
- 用一个共享页面承载越来越多的主题布局分支却不拆配置层

---

## 9. 一句话总结

当前最优主题开发方案是：

`registry 管视觉 token -> manifest 管接线 -> registry loader 管变体注册 -> 业务页优先共享 -> 差异过大再拆独立实现`

后续新增主题时，优先补 token 和 manifest；只有真正需要独立布局时，才新增页面文件。
