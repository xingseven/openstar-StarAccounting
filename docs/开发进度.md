### V2.3.3：修复账单导入重复数据统计Bug (2026-03-22)

- **导入统计准确性提升**：
  - **问题分析**：在执行支付宝/微信账单导入时，如果数据在数据库中已存在（由全局唯一的 `orderId` 决定），`Prisma.createMany` 会自动静默跳过重复数据并返回实际插入条数 0，但这部分未被 `duplicateCount` 统计，导致前端显示“重复 0 条”。
  - **修复方案**：去除了按 `accountId` 过滤的重复性校验，直接对全局唯一字段 `orderId` 进行全局比对；同时，将 `Prisma` 因重复约束自动跳过的数据量（`dbSkippedCount`）准确累加到 `duplicateCount` 之中。
  - **效果**：再次导入已存在的账单时，不再提示迷惑性的“成功 0 条，重复 0 条”，而是准确提示实际的重复条数（例如“重复 158 条”）。

### V2.2.6：重构加载状态与彻底解决滚动恢复 (2026-03-20)

- **问题复盘**：发现即使用 `min-h-screen` 撑起了高度，并且在组件内补全了骨架屏，由于 `page.tsx` 中使用了 Client Component 的 `useState` 和 `useEffect` 控制 `loading` 状态，页面在刷新瞬间依然会有一帧空白（此时 React 还没完成初次渲染与状态初始化），这极短的空白足以让浏览器的 Scroll Restoration 机制判断页面高度为 0，从而强制滚回顶部。
- **最佳实践重构**：
  - 将原本写死在 `consumption/page.tsx` 里的 `SkeletonLoading` 组件提取到了标准的 Next.js App Router 约定文件 `app/(dashboard)/consumption/loading.tsx` 中。
  - **核心原理**：`loading.tsx` 是由 Next.js 服务端直接注入并随初始 HTML 返回的（基于 React Suspense），因此浏览器在收到 HTML 解析的那一刻，页面就已经是撑满骨架屏的完整高度，完美契合了原生浏览器的滚动恢复机制。
  - 移除了上一版本临时加在各个容器上的 `min-h-screen` hack 写法，回归最纯粹的流式渲染架构。

### V2.2.5：彻底修复刷新滚动位置丢失问题 (2026-03-20)

- **根因分析**：虽然 V2.2.4 补全了所有骨架屏组件，但由于 `next/dynamic` 的异步渲染机制以及页面最外层容器（`page.tsx` 和 `ConsumptionDefaultTheme.tsx`）缺乏最小高度限制，在浏览器尝试恢复滚动位置（Scroll Restoration）的瞬间，DOM 的高度依然是 0 或不够，导致滚动条被强制重置到了顶部。
- **解决方案**：
  - 为 `consumption/page.tsx` 的最外层容器添加了 `min-h-screen` 类。
  - 为 `SkeletonLoading` 组件的最外层容器添加了 `min-h-screen` 类。
  - 为 `ConsumptionDefaultTheme` 的主内容区容器添加了 `min-h-screen` 类。
- 确保在 React hydration 和数据请求的任何阶段，页面都有足够的高度支撑浏览器的滚动条恢复。
