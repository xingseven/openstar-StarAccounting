# OpenStar 后端目录收口开发文档

## 0. 文档版本

- 大版本：`v1.0`
- 小版本：`v1.0.0`
- 本次更新：将唯一后端工程从 `src/server/` 收口到仓库根目录 `server/`，统一脚本、工作流、部署入口与运维文档。

---

## 1. 背景

前端目录完成收口后，仓库里仍保留 `src/server/src` 这一层历史结构。虽然功能上只剩一个后端工程，但路径形态仍容易引起两类误解：

1. 新成员会误以为 `src/` 下仍存在一套业务源码主入口。
2. 根脚本、CI、Docker 与部署脚本继续引用 `src/server/`，会让目录结构与实际职责不一致。

本轮整理的目标不是改后端业务逻辑，而是把仓库结构收口成更直观的 `web/ + server/` 双入口。

---

## 2. 目标

1. 将唯一后端工程迁移到根目录 `server/`。
2. 保持现有后端源码、Prisma、测试、构建产物和部署脚本可继续使用。
3. 统一所有当前仍生效的入口与文档说明，避免仓库继续出现 `src/server/` 作为现行路径。

---

## 3. 影响范围

本次收口涉及以下范围：

1. 根目录脚本：`package.json`
2. 容器与工作流：`docker-compose.yml`、`.github/workflows/ci.yml`、`.github/workflows/deploy-non-docker.yml`
3. 后端部署脚本：`server/scripts/deploy-linux.sh`、`server/scripts/deploy-windows.ps1`
4. 当前运维文档：`README.md`、`docs/数据库配置.md`、`docs/数据库结构核对报告.md`、`docs/非Docker自动部署开发文档.md`
5. 版本记录：`CHANGELOG.md`、`docs/开发进度.md`

---

## 4. 执行步骤

### 4.1 目录迁移

1. 将 Git 已跟踪的后端文件从 `src/server/` 迁移到 `server/`。
2. 保留 `package.json`、`package-lock.json` 中已有的未提交改动，不做回滚。
3. 将本地 `.env` 与运行依赖同步到新目录，确保迁移后仍可直接在 `server/` 下执行命令。

### 4.2 引用收口

1. 将根目录 `npm run dev` 的后端入口改为 `cd server && npm run dev`。
2. 将 Docker Compose、CI、非 Docker 部署工作流统一改为引用 `server/`。
3. 将 Linux / Windows 部署脚本中的工作目录、帮助文案和路径校验统一改为 `server/`。

### 4.3 文档同步

1. 更新根目录 `README.md` 中的仓库结构说明。
2. 更新数据库、部署和迁移文档中的后端路径。
3. 新增本开发文档，补充这次目录收口的背景、范围与验收标准。

---

## 5. 风险与处理

### 5.1 本地运行进程占用旧目录

Windows 下如果仍有开发进程以 `src/server/` 为工作目录，旧空目录可能暂时无法立即删除。此问题不影响 Git 路径迁移与新目录 `server/` 的后续使用，但需要在释放进程后完成本地清理。

### 5.2 脚本入口遗漏

本轮的主要风险不在代码逻辑，而在路径引用遗漏。因此验收重点放在：

1. 根脚本是否能定位到 `server/`
2. CI / 部署工作流是否仍引用旧路径
3. 运维文档是否仍指导使用 `src/server/`

---

## 6. 验收标准

满足以下条件即可视为收口完成：

1. 仓库内唯一后端工程路径为 `server/`
2. 根目录脚本、Docker、CI、非 Docker 部署链路均指向 `server/`
3. 当前仍会被使用的文档不再将 `src/server/` 作为现行路径
4. `npm.cmd --prefix server run build` 通过
5. `npm.cmd run build` 通过
